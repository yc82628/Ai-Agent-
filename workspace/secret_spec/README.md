# Hidden Task Release

Welcome to the actual challenge.

This folder contains the task materials released at **20:00 on Thursday**. Before now, you were building the agent. From this point on, your agent should read the specification, implement the required program, run public tests, inspect failures, patch the solution, and keep logs of what happened.

In other words: this is where the “agent” part stops being a nice word in a README and starts having responsibilities. Tragic, but educational.

## What is in this folder

```text
RELEASE_2000/
  SECRET_SPEC.md
  public_tests/
  expected_outputs/
  test_runner/
```

### `SECRET_SPEC.md`

The full hidden task specification.

This is the source of truth for the required behavior. Read it carefully. Better yet, make your agent read it carefully, because that is rather the point of the event.

### `public_tests/`

The public test inputs and metadata.

These tests are participant-facing and are meant for iteration. They are grouped by topic so your agent can work through failures category by category instead of flailing heroically in every direction at once.

See:

```text
public_tests/README.md
```

### `expected_outputs/`

Reference expected outputs for the public tests.

These were generated by organizers using the official reference implementation.

You do **not** need to generate these files. You should not generate these files from your own program. That would be like grading your exam with an answer key you wrote yourself. Bold, useless, spiritually concerning.

See:

```text
expected_outputs/README.md
```

### `test_runner/`

The public test runner.

This is the tool you use to run public tests against your implementation and get a scoreboard.

See:

```text
test_runner/README.md
```

## First steps after the reveal

Recommended order:

1. Read `SECRET_SPEC.md`.
2. Start or restart your agent with the hidden spec in context.
3. Let the agent create an implementation plan.
4. Run the public test suite.
5. Inspect failing categories.
6. Patch the implementation.
7. Rerun tests.
8. Repeat.
9. Keep logs the entire time.

The goal is not a one-shot generation. The goal is an implementation-test-repair loop that keeps improving.

## Basic public test command

From the repository root, run:

```bash
python3 RELEASE_2000/test_runner/run_tests.py --compiler "python3 knit.py"
```

If your implementation entrypoint is somewhere else, replace the compiler command:

```bash
python3 RELEASE_2000/test_runner/run_tests.py --compiler "python3 path/to/your/solution.py"
```

Run one public level:

```bash
python3 RELEASE_2000/test_runner/run_tests.py --compiler "python3 knit.py" --category level_03_brackets
```

Limit printed failures:

```bash
python3 RELEASE_2000/test_runner/run_tests.py --compiler "python3 knit.py" --failures 5
```

Smoke mode:

```bash
python3 RELEASE_2000/test_runner/run_tests.py --compiler "python3 knit.py" --mode exit-only
```

Smoke mode checks exit codes and basic stdout discipline only. It is useful for sanity checks, not full correctness scoring. Please do not emotionally attach yourself to a smoke score. It is named smoke for a reason.

## Public tests vs hidden tests

Public tests are for iteration. Use them often.

Hidden tests are for final judging. They are not included in this folder.

Hidden tests:

- are based on the same specification;
- use the same comparison principles;
- do not test behavior outside the spec;
- may combine rules more aggressively than the public tests;
- are intended to measure generalization, not memorization.

The hidden test score counts as **40% of the final score**. Treat it like an additional judge: quiet, unforgiving, and immune to your presentation charisma.

## Expected outputs

For full public scoring, the runner needs expected outputs.

Those expected outputs are already included in this release package under:

```text
expected_outputs/
```

Participants should not run any expected-output generation script.

If expected outputs are missing or the runner complains about missing expected files, ask in Slack `#public-tests`.

Do not generate expected outputs from your own implementation. That is not testing. That is a mirror with a command-line interface.

## Logs still matter

While your agent works, keep the required logs updated:

```text
agent_logs/
  prompts.log
  decisions.log
  commands.log
  test_runs.log
  errors.log
  human_interventions.log
  final_report.md
```

Use the templates in the main repo if you have not created your own.

Judges will look at the final program, but also at how it was produced: prompts, commands, test runs, failures, human interventions, and score progression.

A final program with no process evidence is not a strong submission. It is just a mysterious file with confidence issues.
